# Microsoft Graph with PowerShell 101

## Abstract

In this follow-along, we will explore Graph automation by creating a Microsoft Entra app registration, an Azure Automation runbook to check your other Entra apps for expired certificate or credential secrets. All purely PowerShell, of course.

This session is ideal for people new to Entra and Graph, but also for intermediate users. We will explore the differences between Enterprise Apps and App Registrations, Delegated and Application permissions as well as authentication flows.

To make the most of this follow-along, be sure to create your own, free Entra tenant to try out all that stuff. If you cannot create a tenant, credentials with a very minimal set of permissions will be given to you, and your experience will be a little less interesting.

## App Registrations and Enterprise Applications

To be able to authenticate with a specific set of permissions, either interactively or non-interactively,
we always need an Application in Entra. Does it have to be your tenant, though? Let's see:

- Application
   - Describes how to connect
   - Describes which permissions are available
   - Describes its own roles and scopes
   - Specifies credentials to connect, if required
   - Lives in the remote tenant (!) that is offering e.g. a SaaS product
- Enterprise Application (Service Principal)
  - Describes who is allowed to connect
  - Describes if and how the Application is available in the home tenant
  - Lives in the home tenant that consumes a resource from the remote tenant
  - Also created for each System and User-Assigned Managed Identity

So, without further ado, let's start!

1. The API resource we're using is the [application](https://learn.microsoft.com/en-us/graph/api/resources/applications-api-overview?view=graph-rest-1.0)
1. In order to `Create` an application, we can use...

   <summary>Which method?</summary>
   Create: <https://learn.microsoft.com/en-us/graph/api/application-post-applications?view=graph-rest-1.0&tabs=http>
   </details>
1. Sign-in to Azure, using `Connect-AzAccount`
1. Sign-in to Graph, using `Connect-EntraService -Service Graph -AsAzAccount`
1. Build the app body. The app should have the display name "PSConf Anniversary: Graph 101"
   <details>
   <summary>Hint</summary>
   Explore the API resource <https://learn.microsoft.com/en-us/graph/api/resources/application?view=graph-rest-1.0>

   Our Body thus becomes

        displayName = "PSConf Anniversary: Graph 101"
    }

1. To be able to sign-in to an Application using the interactive Token Auth (Browser) flow, we need to configure the
   authentication platform. Our application, when used interactively, acts as a public client. Thus, we

   <details>
   <summary>Hint</summary>

## Scopes and Roles

To be able to use the app for anything losely related to automation, we should add the roles/scopes we require. Why
the distinction, again?

OAuth2 Permission Scopes describe the delegated permissions available to interactive users.
Roles describe application roles available to non-interactive users, i.e. applications.

So, where do we get started while looking for resources? Well, the authoritative source
is always best: <https://learn.microsoft.com/en-us/graph/permissions-reference>

That being said, the [Graph X-Ray extension](https://graphxray.merill.net/) is an excellent
start if you want to migrate away from ClickOps. Remember the script export feature in various
Microsoft tools? Yeah - Graph X-Ray will give you that.

The other tool that can help explore the Graph API is [Graph Explorer](https://developer.microsoft.com/en-us/graph/graph-explorer).
It includes some sample calls with sample data, but can be connected to your own tenant.

Too much UIs are bad for you, so let's get back to the CLI!

1. Examining the application we registered earlier, it exposes the property `requiredResourceAccess`. Following
   the API doc for [the property](https://learn.microsoft.com/en-us/graph/api/resources/requiredresourceaccess?view=graph-rest-1.0) and
   the [resourceAccess collection](https://learn.microsoft.com/en-us/graph/api/resources/resourceaccess?view=graph-rest-1.0) we can see
   that we need the resource object ID. But what is the resource?

   The resource refers to the service principal in your home tenant. While the Graph Application always has the client ID `00000003-0000-0000-c000-000000000000`,
   the service principal is unique in your tenant.
1. We start by finding the resource ID, that is: The service principal in our tenant. Since we know the app ID, we can query the
   servicePrincipal API:

   ```powershell
   $graphAppId = '00000003-0000-0000-c000-000000000000'
   $graphServicePrincipal = Invoke-EntraRequest -Path "servicePrincipals(appId='$graphAppId')"
   ```
1. To programatically look for scopes and roles without ever leaving PowerShell, examine the properties `oauth2PermissionScopes` and `appRoles`!
   
   ```powershell
   $graphServicePrincipal.oAuth2PermissionScopes
   $graphServicePrincipal.appRoles
   ```
1. Our delegated permission has to do with Hello and Authentication. Let's have a look!
   
   ```powershell
   # Delegated - Meaning: Scope!
   $graphServicePrincipal.oAuth2PermissionScopes | Where value -match Hello
   ```
1. The previous command should return 4 scopes. In our help desk scenario, `UserAuthMethod-WindowsHello.ReadWrite.All` sounds like a good fit. Notice
   that the id of the permission is exactly the same as in the Graph API docs. Use it to prepare the first entry for our required resource access hash!
   
   ```powershell
   # Delegated - Meaning: Scope!
   $delegate = $graphServicePrincipal.oAuth2PermissionScopes | Where value -eq UserAuthMethod-WindowsHello.ReadWrite.All
   $body = @{
      displayName            = "PSConf Anniversary: Graph 101"
      publicClient           = @{
         redirectUris = @(
               'http://localhost'
         )
      }
      requiredResourceAccess = @(@{
         resourceAppId  = $graphServicePrincipal.appId
         resourceAccess = @(
               @{
                  id   = $delegate.id
                  type = "Scope"
               }
         )
      })
   }
   ```
1. Let's do something very similar to find our application permission. The app should export expired app secrets in an unattended job, leading
   to `Application.Read.All`.

   ```powershell
   $role = $graphServicePrincipal.appRoles | Where value -eq Application.Read.All
   $body.requiredResourceAccess[0].resourceAccess += @{
      id   = $role.id
      type = "Role"
   }
   ```
1. Your table should look similar to:

   ```json
   {
   "requiredResourceAccess": [
      {
         "resourceAccess": [
         {
            "type": "Scope",
            "id": "13eae17d-aaa4-47b8-aaee-0eb33c6e2450"
         },
         {
            "type": "Role",
            "id": "9a5d68dd-52b0-4cc2-bd40-abcf44ac3a30"
         }
         ],
         "resourceAppId": "00000003-0000-0000-c000-000000000000"
      }
   ],
   "publicClient": {
      "redirectUris": [
         "http://localhost"
      ]
   },
   "displayName": "PSConf Anniversary: Graph 101"
   }
   ```
1. With that done, we can update the application using the HTTP-method `PATCH` and the application's object id or application id:
   
   `Invoke-EntraRequest -Method Patch -Path applications/$graph`
1. Either GET the app again or view the portal to verify that the app permissions have indeed been requested and not yet been consented to.
1. Congratulations! You're well on your way to actually automate some things using the Graph API!

## Consent

Consent in Entra ID comes two-fold, admin consent and user consent, and describes the state of your application as in: Has
it been approved to use its requested scopes/roles?

If admin consent is not required, for example for `User.Read`, the user may consent themselves. This also depends on your tenant's settings.
For other, more far-reaching permissions, admin consent may be required, like for our delegated permission `UserAuthMethod-WindowsHello.ReadWrite.All`.

In our case, both permissions require admin consent.

1. We start with the easy one, granting app role assignments. Looking at [the docs](https://learn.microsoft.com/en-us/graph/api/serviceprincipal-post-approleassignments?view=graph-rest-1.0&tabs=http) tells
   us that we need three pieces of information, the resource id, principal ID and app role id.
   
   The resource id in our case is the Graph **Service Principal** since we require resource access in our home tenant.
   The principal id is the object id of our applications' service principal
   The app role id is what we already retrieved in the last section
1. With this information, we can start building our request:

   ```powershell
   $appServicePrincipal = Invoke-EntraRequest -Path "servicePrincipals(appid='cdc0925b-d7eb-49c8-bc92-980dc5da44b2')"
   $graphApp = '00000003-0000-0000-c000-000000000000'
   $graphServicePrincipal = Invoke-EntraRequest -Path "servicePrincipals(appId='$graphApp')"
   $role = $graphServicePrincipal.appRoles | Where value -eq Application.Read.All
   $roleAssignment = @{
      principalId = $appServicePrincipal.id
      resourceId  = $graphServicePrincipal.id
      appRoleId   = $role.id
   }
   ```
1. The request is a POST, since we are creating new objects:

   `Invoke-EntraRequest -Method Post -Path servicePrincipals/$($appServicePrincipal.id)/appRoleAssignments -Body $roleAssignment -ContentType application/json`

1. Excellent. Now for the second item on the agenda: Granting delegated consent for all users in the tenant! Searching for grants in the docs should
   eventually point you to [the docs](https://learn.microsoft.com/en-us/graph/api/oauth2permissiongrant-post?view=graph-rest-1.0&tabs=http).

   The consent type `AllPrincipals` does what it says on the tin: Grant consent for all users in the tenant! To use the consent type `Principal`, you
   would also specify a principal id, for example the object id of a user.

   The scope, this time, is a space-separated list of strings. Oh boy! The developers of this API endpoint did not like arrays, it seems.
   
   ```powershell
   # lacking documentation...
   $delegate = $graphServicePrincipal.oAuth2PermissionScopes | Where value -eq UserAuthMethod-WindowsHello.ReadWrite.All
   $oauth2PermissionGrant = @{
      resourceId  = $graphServicePrincipal.id
      consentType = 'AllPrincipals'
      clientId    = $appServicePrincipal.id
      scope       = $delegate.value -join ' ' # Or trust the $OFS
   }
   ```
1. This request goes to... the oauth2PermissionGrants endpoint! Contrary to what you would guess, this endpoint is not
   scoped to a service principal, but is an organization-wide resource.

   `Invoke-EntraRequest -Method Post -Path oauth2PermissionGrants -Body $oauth2PermissionGrant -ContentType application/json`
1. Awesome work! With all those steps done, we can **actually** use our app! Try signing in to it using `Connect-EntraService`!
   <details>
   <summary>Hint</summary>
   The Parameter `Path` expects an API path relative to the API source.
   In our case this is `application`.
   Reading the docs or guessing informs us that the method to use is `POST` - not unusual with REST APIs.
   The full call would be `$app = Invoke-EntraRequest -Path applications -Method Post -Body $body -ContentType application/json`
   </details>
1. Try calling an endpoint you should not be able to access, such as `groups`

   `Invoke-EntraRequest -Path Groups`
1. Now for something more tricky: The application has the API permission to read all applications. Can you read all applications
   as a user?

   `Invoke-EntraRequest -Path applications`

## All the secrets

The third component of our journey towards a help desk app are credentials! Why, though? We were already able to sign-in
to the app, weren't we? Well: Interactively, yes. To be of use to us in automation tasks that are running non-interactively or
to be more specific may not be interrupted by MFA prompts and CA policies, we create secrets!

Logging into an app using a secret unlocks a different set of permissions for us, the application permnissions. So, after creating a
secret, you can sign-in as the application itself.

>NOTE: While we will create all three credential types, in practice you should try use use managed identities tied to
>resources or federated credentials tied to external identity providers such as Azure DevOps or GitHub as
>much as possible! Certificates and passwords get leaked, federated credentials can mitigate that risk.

## OData Query parameters

Not all resources on the Graph are created equally. The API endpoints we selected
so far support multiple [query parameters](https://learn.microsoft.com/en-us/graph/query-parameters?tabs=http). For
this follow-along we're going to take the arguably most important ones as examples.

Why do we need to use these parameters, anyways? Why don't we simply use `Where-Object` to filter?
The reason is the same as with other data sources such as Active Directory or SQL: Retrieving all objects
to filter them later on is an unnecessary strain on the source. Think one step further: What if your API is rate-limited or you
pay per number of objects received? The old PowerShell mantra of "Filter early" is still true.
